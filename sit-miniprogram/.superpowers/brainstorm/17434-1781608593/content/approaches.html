三种架构方案

选一个最适合这个项目的起点

A

单页极简（快速 MVP）

所有逻辑写在一个 Page 里：VisionKit 检测、计时器、提示、统计全部内联。底部 TabBar 切换「检测 / 统计 / 勋章」三个页面。

pages/
├── monitor/ ← VisionKit + 检测逻辑 + 计时
├── stats/ ← 读 Storage 展示图表
└── badges/ ← 勋章列表

✅ 优点

  • 开发最快（2–3周可完成）
  • 代码量少，容易 debug
  • 没有多余抽象

⚠️ 缺点

  • monitor 页面会变得很大
  • 后期加云同步要大改
B

服务层分离（推荐 ⭐）

抽出 PostureService（检测逻辑）+ SessionStore（计时 & 统计）两个独立模块，页面只负责展示。加云同步时只需改 Store，页面不动。

services/
├── PostureService.js ← VisionKit + 判姿逻辑
├── SessionStore.js ← 计时、统计、勋章触发
└── AlertService.js ← 语音/震动/静音策略
pages/
├── monitor/ stats/ badges/ settings/

✅ 优点

  • 逻辑与 UI 解耦，易于测试
  • 后期加云同步只改 Store 层
  • 多页面共享同一检测状态

⚠️ 缺点

  • 比方案 A 多 1 周开发量
  • 需要理解模块化设计
C

Worker 线程分离（过度设计）

把姿态推理放到 Worker 线程，UI 线程只处理渲染，理论上最流畅。但微信小程序 VisionKit 本身不支持在 Worker 中调用，需要折中方案。

workers/
└── postureWorker.js ← 仅计算，不含 VisionKit
pages/
└── monitor/ ← VisionKit 仍在主线程

✅ 优点

  • 理论上 UI 更流畅

⚠️ 缺点

  • VisionKit 无法在 Worker 运行
  • 实际收益有限
  • 调试复杂度翻倍
💡 我的推荐是 B：服务层分离，既不过度设计，又为你选择的「先本地后云同步」预留了干净的扩展点。
